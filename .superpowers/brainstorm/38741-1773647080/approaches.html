字符串提取：3 种技术方案

选择核心提取策略，影响性能、准确性和实现复杂度

A

Phase2 集成扫描（推荐）

原理：在 phase2 构建内存索引时，同步维护一个字节级内存镜像 (HashMap<u64, u8>)。每次 WRITE 操作后，检查该区域是否形成/修改了可打印字符串。

优点：

  • 单次扫描，零额外 I/O
  • 天然按时序记录字符串的出现和覆写
  • 与 Phase2 的 bincode 缓存集成，二次加载秒开

缺点：

  • 增加 Phase2 构建时间（约 5-10%）
  • 内存镜像占用额外内存（通常 <50MB）

性能

⚡ 最优 — 无重复扫描

复杂度

中等 — 需修改 phase2.rs

B

Phase3 后处理

原理：Phase2 完成后，新增一个 Phase3 阶段。遍历 MemAccessIndex 的所有 WRITE 记录，按时序重建内存镜像并提取字符串。

优点：

  • 模块化好，不修改现有 phase2 代码
  • 可独立缓存字符串索引
  • 易于调试和测试

缺点：

  • 需要额外遍历所有内存记录（重复工作）
  • MemAccessIndex 当前无公开迭代器，需暴露内部 HashMap
  • 加载时间增加一个阶段

性能

⚠️ 中等 — 额外一轮扫描

复杂度

低 — 完全独立模块

C

按需懒加载

原理：用户打开字符串面板时，才实时扫描 MemAccessIndex 提取字符串。结果缓存在内存中。

优点：

  • 不影响文件加载速度
  • 不用的功能零开销

缺点：

  • 首次打开面板时有明显延迟（大 trace 可能数秒）
  • 无法参与 bincode 缓存，每次重新打开文件都要重算
  • 用户体验不佳 — 期望即时响应

性能

❌ 差 — 每次需实时计算

复杂度

低 — 完全独立
