反编译功能：三种实现方案

基于需求总结：本地模式匹配引擎 + LLM 增强 + 三栏布局 + CallTree 函数边界

需求回顾

  • 数据来源：Frida Stalker / unidbg trace，不依赖原始 SO 文件
  • 布局：右侧新增反编译栏，支持浮出/关闭
  • 函数边界：CallTree（BL/BLR/RET），支持切换同函数不同调用实例
  • 本地引擎：尽量覆盖完整 ARM64 指令集
  • LLM 增强：用户自配 API Key（OpenAI/Claude/Ollama 等）
A

方案 A：自研 Rust 提升引擎（推荐）

在 Rust 后端实现一个 ARM64 → 伪代码的提升引擎（lifter），直接从 trace 的反汇编文本解析指令语义。

核心流程
1. 指令解析 — 复用现有 taint/parser.rs，扩展为完整的指令语义模型
2. 中间表示(IR) — 将 ARM64 指令转为 SSA 形式的 IR（赋值、load/store、分支、调用）
3. 优化 Pass — 死代码消除、表达式折叠、条件分支→if/else、循环检测
4. 代码生成 — IR → C 风格伪代码文本，附带地址→行号映射

优势

  • 性能最好，纯 Rust 毫秒级响应
  • 复用现有 parser/taint 大量代码
  • 可利用 trace 的运行时数据（真实寄存器值、内存内容）生成带注释的伪代码
  • 不依赖外部工具，零部署成本
  • 地址映射精确（每条 IR 追踪来源指令）

挑战

  • 工程量最大（需实现 IR + 优化 Pass + 代码生成）
  • ARM64 指令集庞大（~500+ 指令），完整覆盖需持续迭代
  • 缺少未执行分支，伪代码不完整（LLM 可补偿）
B

方案 B：Keystone 重汇编 + Ghidra P-Code

用 Keystone 汇编器将反汇编文本重新编码为机器码字节，构造一个"虚拟 ELF"，然后调用 Ghidra 的 Sleigh/P-Code 引擎反编译。

核心流程
1. 重汇编 — Keystone 将反汇编文本 → ARM64 机器码字节
2. 构造虚拟 ELF — 按 SO 偏移地址排列字节，生成最小 ELF
3. Ghidra Headless — 调用 analyzeHeadless 反编译虚拟 ELF
4. 结果解析 — 提取伪代码 + 地址映射，缓存到 bincode

优势

  • Ghidra 反编译质量成熟
  • ARM64 指令覆盖率高（Ghidra 的 Sleigh 定义完整）
  • 不需要原始 SO——从 trace 重建字节

挑战

  • 重汇编可能丢失信息（PC 相对地址、重定位需手动修复）
  • 虚拟 ELF 只有执行过的指令，函数体不完整，Ghidra CFG 分析会受影响
  • 依赖 Ghidra + JVM，部署体积大（~500MB+）
  • Headless 分析慢（秒级到分钟级）
  • 地址映射需要额外处理（Ghidra 行号 → trace seq）
C

方案 C：Capstone + 轻量 IR 混合（折中）

用 Capstone 反汇编库解析指令语义（而非自己写 parser），得到结构化指令信息后，用轻量级 IR 做提升。

核心流程
1. Keystone 重汇编 — 反汇编文本 → 4 字节机器码
2. Capstone 解码 — 机器码 → 结构化指令（操作数、读写寄存器、类型）
3. 轻量 IR + 优化 — 类似方案 A 但 Capstone 省去手写 parser
4. 代码生成 — IR → C 风格伪代码

优势

  • Capstone 的指令语义解析比手写更完整更可靠
  • 不依赖 JVM，Capstone 是轻量 C 库（~1MB）
  • 性能接近方案 A
  • 指令覆盖率高（Capstone 支持完整 ARM64）

挑战

  • 多了 Keystone→Capstone 的中间步骤（反汇编→机器码→再解码）
  • Keystone 重汇编仍有 PC 相对地址问题
  • 需引入 C 库的 Rust binding（capstone-rs / keystone-rs）
  • IR 和代码生成仍需自研

我的推荐：方案 A

理由：你的项目已经有了优秀的 ARM64 指令 parser（taint/parser.rs + taint/types.rs，2000+ 行）和指令分类（insn_class.rs）。 在此基础上扩展 IR 和代码生成，工程量比从零开始小很多。而且 trace 场景下最大的优势——运行时数据——只有自研引擎才能充分利用 （比如直接在伪代码里标注 v0 = *ptr /* =0x16e80, ptr=0x41688658 */）。 Ghidra/Capstone 路线反而增加了不必要的依赖，且无法利用运行时信息。
